- 토스 : **‘Kafka 이중화’** 구성으로 인한 문제점
    - **Active-Active**: 시스템 이중화 방식 중 하나, 두 개 이상의 시스템 혹은 데이터 센터가 모두 활성화되어 동시에 작업을 처리하는 구조.
        - 장점
            - 한 곳에 장애가 발생해도 다른 곳에서 무중단으로 서비스를 지속가능(가용성 매우 높음)
            - Producer와 Consumer는 지역성이 보장된 Kafka에 접근하기 때문에 latency가 상대적으로 짧음
        - 왜 kafka 이중화로 구성했는가?
            - ‘데이터센터 장애 발생 시 전체 서비스 통신 마비’ 대안으로!!!!
                
                양 클러스터가 독립적으로 구성됐으므로 한쪽 데이터센터 장애가 다른 쪽 클러스터에 영향을 주지 않음
                
            - 완벽한 서비스 이중화를 위해서는 Kafka 클러스터의 이중화가 필수적
    - **해결 과제1) 수백 개의 토픽 데이터를 양쪽이 똑같이 가지도록 복제하는 '데이터 미러링'**
    - **해결 과제2)장애 발생 시 끊긴 부분부터 정확히 이어서 처리하기 위한 'Consumer Group Offset Sync'**
        - 문제점 : 네트워크 지연 때문에 도착 순서가 달라지면 클러스터의 저장 순서가 엇갈릴 수 있음.

    2. 모니터링 방식 :
        
        데이터 파이프라인과 서비스가 안정적으로 운영되는지 시스템 내에서의 모든 지표를 실시간으로 시각화하여 모니터링
        
        - Kubernetes : 쿠버네티스 기반 환경에서 실행 중인 시스템의 모든 상태를 파악
        - `Prometheus` : 쿠버네티스 환경에서 동작하는 앱들의 다양한 지표(메모리 사용 등)을 주기적으로 수집하여 데이터로 저장해주는 시스템
        - `Grafana` : Prometheus에 저장된 데이터를 가져와서 실시간 시스템 상태를 시각화하여 확인할 수 있게 해주는 대시보드
