    - zcalability
    - availability
    - reliability

# 3.1 데이터 아키텍처란

- 요소
    - 프로세스, 전략, 변경 관리, 트레이드오프 평가
    - 운영 아키텍처 - 무엇을 : 인력, 프로세스/기술과 관련한 필요 기능의 요건
        - 데이터는 어떤 비즈니스 프로세스 지원?
        - 조직은 데이터 품질을 어떻게 관리?
        - 데이터가 생성되는 시점부터 쿼리 가능한 시점까지의 지연시간 → 요구사항은?
    - 기술 아키텍처 - 어떻게 : 데이터 수집, 저장, 변환 및 제공 방법을 개략적으로 설명
        - 시간당 19TB의 데이터를 원천 데이터베이스에서 데이터 레이크로 옮기려면 어떻게 해야할까

#### 우수한 데이터 아키텍처

> 
> 
> 
> 최고의 아키텍처를 추구하기보다는 덜 나쁜 아키텍처를 추구하라
> 
- 아키텍처 : 시스템을 형성하는 중요한 설계 결정
    - 중요한 것은 변경 비용으로 측정된다
- 데이터 아키텍처의 목표 : 기본 수준에서 우수한 아키텍처로 이어지기 위한 중요한 결정들을 내리는 것
- ‘우수한’ ⇒ 유연하고 유지 관리하기 쉽다
    - 유연성 유지
    - 적절한 트레이드오프를 실현
    - 광범위하게 재사용 가능한 공통 구성요소를 사용
        
        → 비즈니스 요건을 충족
        
    - 민첩성
        - 새로운 기술 및 관행에 대응해 진화
        - 서로 긴밀하게 결합 X

> 
> 
> - 운영
>     - 상태를 똑똑하게 관리
>     - 관리형 서비스를 선호
> - 자동화
> - 보안
> - 신뢰
> - 성능 효율성
> - 비용 최적화
> - 지속 가능성

#### 데이터 엔지니어링 아키텍처 원칙

---

1. 공통 컴포넌트를 현명하게 선택해라
2. 장애에 대비해라
3. 확장성을 위한 아키텍처를 설계해라
4. 아키텍처는 리더십이다
5. 항상 아키텍처에 충실해라
6. 느슨하게 결합된 시스템을 구축하라
7. 되돌릴 수 있는 의사결정을 하라
8. 보안 우선순위를 지정하라
9. 핀옵스를 수용해라
    1. 핀옵스 : 클라우드 리소스 사용량을 실태와 비용을 투명하게 파악하고 최적화하는 방법론
