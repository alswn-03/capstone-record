#### Kafka 도입이 오버 엔지니어링인지 판단하는 기준

현재 시스템의 요구 사항을 잘 비교해보자

위에서부터 순서대로 가장 중요한 요소

1. (처리량) - **초당 수십만 건 이상의 이벤트를 처리할 트래픽이 있는가?**
2. (운영의 복잡성) - **분산 환경에 따른 인프라 및 운영 복잡도를 감당할 수 있는가?**
3. (다중 소비자) - 동일한 메시지를 여러 시스템이 소비해야 하는가?
4. (내구성) - 데이터 유실을 허용하지 않는 서비스인가?
5. (확장성) - 트래픽이 향후 10 배 이상 증가할 가능성이 있는가?

특히 처리량 부분에서 납득시킬 수 없다면 다른 메시징 큐 도입(Redis, RabbitMQ...)을 고려해보는 것이 좋다.

#### 2. 카프카, 이럴 땐 쓰지 마세요 (Anti-Patterns)

카프카는 설정이 복잡하고 운영 비용이 비쌉니다(주키퍼, 브로커 등). 벼룩 잡는 데 초가삼간을 태울 필요는 없겠죠?

- ① 소규모 작업 큐 (Simple Task Queue)
    
    하루 데이터가 몇천 건 수준이거나, 단순히 A에서 B로 작업을 넘기는 용도라면 카프카는 **오버 엔지니어링(Over-engineering)**입니다.
    
    - **대안**: RabbitMQ, ActiveMQ, 혹은 AWS SQS 같은 가벼운 큐를 사용하세요.
- ② 즉각적인 응답이 필요한 통신 (Request/Reply)
    
    사용자가 버튼을 눌렀을 때 결과를 즉시(동기적으로) 보여줘야 하는 경우입니다. 카프카는 비동기 방식이라 응답이 언제 올지 보장하기 어렵고 구현도 복잡합니다.
    
    - **대안**: gRPC, REST API 같은 직접 통신 방식을 사용하세요
- ③ 장기 데이터 저장소 (Long-term Storage)
    
    카프카가 디스크에 저장한다고 해서, 이를 데이터베이스(DB)나 하둡(HDFS)처럼 쓰면 안 됩니다.
    
    - **이유**: 카프카는 인덱스 기능이 약해서 특정 데이터를 검색(SELECT * WHERE id=...)하기 어렵고, 비용도 비쌉니다.
    - **대안**: 오래된 데이터는 S3나 Data Lake로 옮기세요.

#### 3. 라이벌 비교: Kafka vs RabbitMQ vs Redis vs Pulsar

기술 면접 단골 질문이자, 아키텍처 설계 시 가장 많이 고민하는 부분입니다. 한눈에 비교해 드립니다.

| **특성** | **Apache Kafka** | **RabbitMQ** | **Redis Pub/Sub** | **Apache Pulsar** |
| --- | --- | --- | --- | --- |
| **주요 컨셉** | 분산 로그(Log) | 스마트 브로커(Queue) | 인메모리 채널 | 차세대 메시징 |
| **처리량** | **최상 (Millions/sec)** | 중 (Thousands/sec) | 상 (메모리 속도) | 최상 |
| **데이터 보관** | **디스크 (설정 기간)** | 메모리 (소비 후 삭제) | **보관 안 함 (휘발성)** | 디스크 (계층화 저장) |
| **라우팅** | 단순함 (토픽 기반) | **강력함 (Exchange)** | 단순함 | 강력함 |
| **추천 대상** | 대용량/실시간 파이프라인 | 복잡한 라우팅/작업 큐 | 단순 채팅/알림 | 멀티 테넌시/클라우드 네이티브 |
- **RabbitMQ**: "메시지를 A한테는 주고, B한테는 주지 마" 같은 복잡한 규칙이 필요할 때.
- **Redis**: "데이터가 좀 날아가도 되니, 무조건 빠르고 간단하게 채팅방 만들래."
- **Pulsar**: "카프카 운영이 너무 힘들어. 컴퓨팅과 스토리지를 분리하고 싶어." (카프카의 강력한 경쟁자로 떠오르고 있습니다.)
