# 3. 후보들

## 3-1. 트래픽 시나리오🔵

### ① 트래픽 폭주 시나리오 (Load Balancing & Autonomic Computing)

- **상황:** 특정 이벤트(예: 푸시 알림 발송, 인플루언서의 언급)로 인해 평소보다 10배 이상의 유저가 동시에 접속하여 추천 요청(혹은 클릭 로그 발생)을 보냅니다.
- **해결 및 검증:**
    - 문제 : 대규모 트래픽 발생 - `트래픽을 쏴주는 프로그램(부하 테스트 도구)인 **nGrinder`, JMeter, 또는 Locust**를 사용해 가상의 대규모 트래픽을 발생시킵니다.
    - 대처 방안 : 이때 `로드밸런서`가 아래의 2가지를 보여준다
        - 트래픽을 어떻게 분산하는지,
        - 그리고 AWS Auto Scaling 등을 통해 서버가 어떻게 자율적(Autonomic)으로 늘어나며 버텨내는지(Reliability)
        - `ALB(Application Load Balancer)` : **AWS가 제공하는 L7(애플리케이션) 계층 로드 밸런서**
            
            HTTP/HTTPS 트래픽을 URL 경로 및 호스트 이름에 따라 여러 대상 그룹(EC2, 컨테이너, 람다 등)으로 유연하게 분산하는 핵심 클라우드 네트워킹 서비스
            

### ② 스트리밍 데이터의 병렬 처리와 순서 보장 (Kafka & Ordering)

- **상황:** 사용자의 실시간 행동 로그(클릭, 장바구니 담기, 구매)가 끊임없이 들어옵니다. 파이프라인이 죽지 않고("죽어도 스트림이 안 꺼져") 이 데이터를 처리해야 합니다.
- **해결 및 검증:**
    - 문제 : 트래픽 많아 → **Kafka의 파티션과 컨슈머를 여러 개로 늘려** **`병렬 처리`를 시도 → 순서 역전 문제**
        - 이때, '장바구니 담기' 로그보다 '구매 완료' 로그가 먼저 처리되어버리는 `순서 역전 문제(Ordering Issue)`가 발생하도록 의도적으로 시나리오를 만듭니다.
    - 해결 : 이를 파티션 키(Key) 설정을 통해 동일 유저의 로그는 순서를 보장하도록 해결하는 과정을 데이터베이스에 적재된 결과물로 증명합니다.

## 3-3. 서버가 죽고(가정:약하게 만들어서) → 안전하게 복구 시킬 수 있게

### ④ 시스템 장애 및 인프라 자동 복구 (Terraform & Fault Tolerance)

- **상황:** 예상치 못한 상황으로 데이터베이스 인스턴스나 스트리밍 서버 중 일부가 다운됩니다.
- **해결 및 검증:** 인프라를 수동으로 클릭해서 복구하는 것이 아니라, **HashiCorp Terraform**을 사용해 코드로 인프라(IaC)를 관리하고 있음을 보여줍니다. 노드가 죽었을 때 얼마나 빠르게 동일한 환경을 재구축하고, 파이프라인이 멈췄던 지점부터 데이터를 유실 없이 다시 읽어올 수 있는지(Offset 관리) 테스트합니다.

<aside>

#### "무엇을 죽이고,무엇이 원활하게 돌아가야 하는가?"(장애 복구 시나리오) → 💡[토스의 서비스 이중화](https://app.notion.com/p/36c071c875dc8028aa6bfcba9d9a245b?pvs=21)

---

'약하게 만든다'기보다는 **'의도적으로 전원을 뽑아버리는(Chaos Engineering)'** 테스트에 가까움

- **무엇을 죽이는가?**
    - 데이터를 소비하는 서버(Kafka Consumer가 떠 있는 EC2 인스턴스) 하나를 강제로 종료시킵니다.
    - 혹은 Kafka 클러스터를 구성하는 노드(Broker) 중 하나를 강제로 내립니다.
- **무엇이 원활하게 돌아가야 하는가?**
    - **데이터 무손실:** 서버가 죽어있는 동안에도 유저의 클릭 로그는 계속 들어옵니다. 이 데이터가 증발하지 않고 Kafka 큐에 안전하게 쌓여 있어야 합니다.
    - **자동 복구 및 이어달리기:** A 서버가 죽으면, 같은 그룹에 있던 B 서버가 A의 몫까지 데이터를 대신 처리하기 시작합니다(이것을 **리밸런싱, Rebalancing**이라고 합니다). 동시에 Terraform이나 Auto Scaling 설정에 의해 죽은 A 서버를 대체할 새로운 C 서버가 자동으로 띄워져야 합니다.
    - **정확한 시점 복구:** 새로 띄워진 서버는 파이프라인이 멈췄던 정확한 시점(Offset)부터 데이터를 중복이나 누락 없이 다시 읽어오기 시작해야 합니다.
</aside>

## 3-4. ‘ML 모델이 포함된 파이프라인’ 특징을 활용하는 시나리오

### ⑥ ML 모델이 포함된 데이터 엔지니어링 시나리오 (MLOps 관점)

모델의 정확도(Accuracy)가 아니라, "모델을 실제 서비스에 어떻게 올릴 것인가(Serving)"에 집중하면 훌륭한 엔지니어링 과제가 나옵니다.

- **시나리오: ML 추론(Inference) 서버의 병목 현상 해결**
딥러닝/머신러닝 모델은 결과값을 내놓는 데(Inference) CPU/GPU 자원을 많이 쓰고 시간이 걸립니다. 트래픽이 몰리면 ML 서버가 뻗어버립니다.
    - **경험 포인트:** 실시간 요청이 들어올 때마다 ML 모델을 돌리는 대신, 유저의 요청을 **Kafka 큐에 담아두고(비동기 추론)**, ML 서버가 큐에서 데이터를 뭉텅이로 가져가 한 번에 예측(Batch Inference)한 뒤 결과를 Redis에 꽂아주는 아키텍처를 구축합니다. "느린 ML 모델을 대규모 트래픽 환경에서 죽지 않고 서비스할 수 있는 서빙 아키텍처를 만들었다"는 엄청난 강점이 됩니다.
- **시나리오: 실시간 피처 스토어 (Feature Store) 업데이트**
추천 모델이 똑똑해지려면 유저가 '방금 전'에 누른 상품 정보가 바로 모델에 전달되어야 합니다. 카프카를 통해 실시간 로그를 받아 유저의 '실시간 선호도 벡터'를 계산해 Redis에 0.1초 만에 업데이트하고, ML 모델은 이 신선한 데이터를 바탕으로 추천을 내어주는 파이프라인을 구축해볼 수 있습니다.

## 3-5. 빠른 추천 계산 목표로 : 컨퍼런스에서 본 발표 활용해도 좋을 것 같은데 ❌

<aside>

**이 시나리오는 ‘모델의 정확도’도 챙겨야 하는 부분이라 하지 않는게 좋을 것 같음**

---

현재 추천 시스템 분야에서는 **딥러닝 모델의 복잡도(정확도)와 실시간 서빙의 지연 시간(Latency) 사이에 필연적인 충돌**이 발생합니다. 본 프로젝트는 이 트레이드오프를 해결하기 위해, 무거운 ML 연산을 Offline(배치)으로 분리하고 Online(실시간) 구간은 Kafka-Flink와 Vector DB를 활용한 **Two-Stage(후보 추출 및 랭킹) 아키텍처로 디커플링하여 지연 없는 실시간 MLOps 환경을 구축**하는 것을 목표

---

'잘못된 파이프라인'을 적극 이용 해서 성능 비교 해보면 어떨까?

1. **V1 아키텍처 (발표에서 본 안티 패턴 의도적 구현):** 일단 그 타팀처럼 무식하게 실시간 파이프라인에 BERT4Rec을 얹어봅니다. 그리고 JMeter 같은 부하 테스트 도구로 트래픽을 쏴봅니다. 초당 100건만 들어와도 서버가 버벅대고 지연율(Latency)이 10초, 20초로 치솟으며 파이프라인이 터지는 것을 직접 수치로 증명(모니터링 대시보드 캡처)합니다.
2. **V2 아키텍처 (정석 아키텍처):**
이 문제를 해결하기 위해 연산을 배치와 스트림으로 분리한 정석적인 Two-Stage 아키텍처를 적용합니다. 똑같이 트래픽을 쐈을 때, 지연 시간이 0.05초로 줄어들고 파이프라인이 안정적으로 돌아가는 것을 보여줍니다.

문제 상황(AS-IS)을 데이터로 증명하고, 아키텍처 개선을 통해 결과(TO-BE)를 수치로 보여주는 스토리

</aside>

        - 진짜 '전형적인 실시간 추천(Two-Stage)'?
            
            빅테크 기업들이 무거운 딥러닝 모델을 실시간에 사용하면서도 지연을 없애는 '진짜' 스탠다드 방식은 다음과 같습니다.
            
            - **새벽(Batch)에 하는 일:** 단순히 인기 상품만 랭킹 매겨서 넣는 것이 아니라, **무거운 딥러닝 모델을 밤새 돌려서 쇼핑몰 내 모든 상품의 수학적 좌표(Embedding Vector)를 뽑아내어 빠르고 가벼운 Vector DB(또는 Redis)에** 미리 싹 다 밀어 넣습니다.
            - **실시간(Stream)에 하는 일:** 유저가 상품을 3번 클릭합니다. 실시간 서버는 무거운 딥러닝 모델을 다시 켜지 않습니다. 방금 유저가 누른 상품 3개의 좌표값을 더한 뒤, Vector DB에 있는 수만 개의 상품 좌표 중 **가장 가까운 것(유사한 것)을 단순 수학 계산(내적, ANN Search)으로 빠르게 찾아옵니다.**

<aside>

**내가 기존에 만들어둔 아키텍처 : 가장 정석적인 실시간 추천(MLOps) 아키텍처**

결론부터 말씀드리면, 새로 보내주신 캡처본(**내가 기존에 만들어둔 아키텍처)**과 제가 설명해 드린 방식은 '완벽히 같은 구조(스탠다드)'이고, 이전에 보셨던 타팀(BOAZ 1조)의 프로젝트 캡처본만 '특이한 병목 구조'를 가지고 있습니다.

세 가지가 머릿속에서 섞이실 텐데, 가장 헷갈리셨던 '새벽 집계(Batch)'와 **'실시간 추론(Online)'** 단계에서 이들이 구체적으로 무엇을 계산하고 서빙하는지 명확하게 비교해 드릴게요.

### 1. 새벽에 하는 일 (Offline / Batch Pipeline)

가장 큰 차이는 "새벽에 얼마나 무거운 연산을 미리 해두는가"에 있습니다.

- **BOAZ 팀 (이전 캡처): 단순 랭킹 집계**
    - 단순히 "어제 20대 여성이 가장 많이 본 상품 Top 10"을 SQL로 집계해서 Redis에 넣어둡니다.
    - 모델과 관련된 계산은 새벽에 하나도 하지 않았습니다.
- **새 캡처본 & 제 설명 (정석): 가장 무거운 ML 연산 처리**
    - 새 캡처본의 왼쪽을 보면 "오프라인에서 계산한 유저/아이템 피처 정보를 Feature Store에 업데이트"라고 되어 있습니다.
    - 이게 바로 제가 말씀드린 "밤새 무거운 딥러닝 모델을 돌려서 수만 개 상품의 벡터(좌표)나 유저의 장기 선호도를 미리 다 계산해 둔다"는 뜻입니다. 이 복잡한 계산 결과물(피처)을 담아두는 바구니가 Feature Store(혹은 Vector DB)입니다.
    - 동시에 오른쪽을 보면 **"새로운 버전의 모델을 Inference Server에 업데이트"** 합니다. 낮 동안 수집된 데이터로 모델을 똑똑하게 재학습시켜서 서버에 갈아 끼워두는 작업입니다.

### 2. 실시간 추천을 할 때 (Online Pipeline)

유저가 방금 상품을 3번 클릭했을 때, 서버가 어떻게 움직이는지 보면 차이가 극명합니다.

- **BOAZ 팀 (이전 캡처): 실시간에 딥러닝 쌩으로 돌리기 (병목 발생)**
    - 미리 계산해둔 '피처(재료)'가 없으니, 유저가 3번 클릭&마지막 클릭까지 30분 뒤에  무겁고 거대한 BERT4Rec 모델 전체를 실시간 서버 위에서 처음부터 끝까지 가동합니다.
    - 이러면 추천 결과가 나오기까지 연산 시간이 너무 오래 걸려 파이프라인 전체가 막힙니다.
- **새 캡처본 & 제 설명 (정석): 이미 있는 피처로 가볍게 추론하기 (준실시간)**
    - 새 캡처본의 아래쪽 상자를 보면 "사용자가 접속한 순간, 가장 최신 상태를 반영해 추천"한다고 되어 있습니다.
    - 유저가 클릭을 하면, 무거운 모델을 처음부터 돌리지 않습니다. 새벽에 **Feature Store**에 빵빵하게 채워둔 '유저 피처(취향)'와 '아이템 피처(벡터)'를 0.01초 만에 쏙 빼옵니다.
    - 이 준비된 재료들을 Inference Server(추론 서버)에 던져주면, 서버는 딥러닝 가중치 곱셈 같은 무거운 짓 대신 "가져온 재료들끼리의 단순 유사도 계산(내적)" 혹은 "아주 가벼운 랭킹 계산"만 찰나의 순간에 수행하여 결과를 반환합니다.

### 💡 요약하자면

새로 보내주신 장표의 'Feature Store'가 제가 말씀드린 'Vector DB (미리 계산된 재료 창고)'의 역할을 하는 곳이고, 'Inference Server'가 '정교한 랭킹이나 유저 인코딩을 가볍게 수행하는 서버'입니다.

실제 서비스에서는 이렇게 연산의 부하를 오프라인(새벽)과 온라인(실시간)으로 쪼개어(Decoupling) 실시간 지연을 방지합니다.

---

> 컨퍼런스 팀에서 ‘실시간 추천 계산’이 오래걸린 이유
> 
> 
> ### 1. Flink '30분 Session Window'의 잘못된 활용 (논리적 지연)
> 
> - **원인:** 실시간 추천의 핵심은 유저가 상품을 **'클릭하는 즉시'** 반응하는 것입니다. 하지만 이 팀은 Flink에서 유저의 행동이 30분 동안 없을 때까지 기다렸다가 세션을 묶어서 처리하는 안티 패턴(Anti-pattern)을 설계했습니다.
> - **결과:** 조기 트리거(예: 3회 클릭 시 방출)를 억지로 끼워 넣었더라도, 근본적으로 스트리밍 엔진이 유저의 데이터를 즉각 방출하지 않고 메모리에 쥐고 대기하는 구조였기 때문에 불필요한 대기 시간이 발생했습니다.
> 
> ### 2. 'Hot Path(실시간 구간)'에 무거운 딥러닝 모델 통째로 배치 (물리적/연산 지연)
> 
> - **원인:** 0.01초가 급한 실시간 데이터 처리 구간(Hot Path)에 연산량이 방대한 트랜스포머 기반 딥러닝 모델(BERT4Rec)을 올려두었습니다.
> - **결과:** 유저가 클릭을 할 때마다 이 무거운 모델의 수많은 레이어를 처음부터 끝까지 통과시키는 생(Raw) 연산을 수행해야 했습니다. 모델 자체가 너무 무거워 한 건을 추론하는 데 오랜 시간이 걸렸고, 유저가 조금만 몰려도 큐에 데이터가 쌓이는 심각한 병목(Backpressure)이 발생했습니다.
> 
> ### 3. 'Feature Store(사전 계산된 재료 창고)'의 부재 (역할 분담 실패)
> 
> - **원인:** 정석적인 아키텍처라면 새벽 배치(Batch) 시간에 무거운 모델을 돌려 모든 상품의 벡터(좌표)값과 유저 피처를 미리 다 계산해서 빠르고 가벼운 저장소(Feature Store/Vector DB)에 쌓아두어야 합니다.
> - **결과:** 이 팀은 새벽 시간에 단순히 '어제 가장 많이 팔린 인기 상품' 정도만 Redis에 넣어두었을 뿐, 정작 무거운 AI 연산은 하나도 미리 해두지 않았습니다. 결국 **미리 해둘 수 있는 무거운 수학 숙제(벡터 추출, 임베딩)를 굳이 실시간 환경으로 끌고 와서 처음부터 다 풀게 만들었기 때문에** 응답 속도가 박살 날 수밖에 없었습니다.
> 
> **💡 한 줄 요약:**
> 결국 이 파이프라인이 느려진 핵심 이유는 "오프라인(새벽)에서 해야 할 무거운 짐을 온라인(실시간) 서버 혼자 다 짊어지게 만들었고, Flink를 실시간 스트리밍이 아닌 지연 버퍼처럼 사용했기 때문"입니다.
> 
> 질문자님의 프로젝트에서는 이 세 가지 실패 원인을 정확히 반대로 뒤집어서(오프라인 사전 계산 + 즉각적인 실시간 검색) 설계하신다면, 면접관이 박수를 칠 만한 완벽한 문제 해결 스토리가 완성될 것
>
