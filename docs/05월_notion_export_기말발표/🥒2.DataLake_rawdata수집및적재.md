# 2. 원시 데이터 수집 및 적재(data lake : raw_data→S3)


<aside>

### ❇️ 보편적인 포맷

**웹 → kafka(버퍼,준실시간 구현) → [S3(영구 백업,datalake) & 추천 엔진(실시간 소비)]**

---

- **S3 (데이터 레이크/영구 저장소):** 거대한 '물류 창고'입니다. 데이터가 한 번 들어가면 평생 보관됩니다. 엄청나게 싸고 무한정 넓지만, 여기서 데이터를 0.1초 만에 쏙쏙 빼서 실시간으로 추천 결과를 계산하기에는 너무 무겁고 느립니다.
- **Kafka / Redis Streams (메시지 브로커/버퍼):** 미친 듯이 돌아가는 '고속 컨베이어 벨트'입니다. 백엔드에서 로그가 쏟아지면 벨트 위에 던져놓고, 추천 엔진은 그 벨트 끝에서 0.1초 만에 데이터를 날렵하게 낚아채서 계산합니다.
    - *주의할 점:* Redis는 데이터를 하드디스크가 아닌 메모리(RAM)에 저장합니다. 벨트(RAM)가 꽉 차면 예전 데이터부터 버려야 하므로, 추천 엔진이 데이터를 가져가서 처리한 뒤에는 진짜 창고인 **S3**로 따로 옮겨서 영구 보관해 줘야 합니다.
</aside>


## 0. 현재 현황

Web에서 view, addtocart, transaction 버튼을 누르면 로그 데이터가 생성되고,

해당 로그 데이터는 JSON 현태로 server 폴더의 logs.json파일에 적재된다

### ✔️ Raw 데이터는 어디에, 어떻게 저장하는가?

**표준적인 실무 데이터 파이프라인 흐름:**

    웹 로그 ➡️ 로그 수집 API 서버 ➡️ Kafka ➡️ (실시간 처리용 DB & 장기 저장용 S3)


1. **Log Collector (API 서버):** 웹에서 보낸 로그를 가장 먼저 받는 가벼운 백엔드 서버(Node.js, FastAPI 등)입니다. 데이터를 받아서 유효성을 검사한 뒤 즉시 Kafka로 밀어 넣습니다.
2. **Kafka (메시지 큐 / 스트리밍 허브):** 로그 데이터가 쏟아져 들어올 때 이를 임시로 버퍼링하고, 필요한 곳에 데이터를 뿌려주는 역할을 합니다.
3. **데이터의 분기 (Lambda Architecture):** Kafka에 들어온 데이터는 두 갈래로 나뉩니다.
    - **실시간 처리 (Speed Layer):** Spark Streaming이나 Flink가 Kafka 데이터를 읽어 실시간 인기 상품을 계산하거나, 사용자 세션 기반 추천을 업데이트하여 Redis 같은 In-memory DB에 저장합니다.
    - **배치 처리 및 적재 (Batch Layer):** Kafka Connect나 Logstash 등을 통해 S3 (Data Lake)에 Raw 데이터를 쌓습니다. 이 데이터는 나중에 무거운 추천 모델(딥러닝, 협업 필터링 등)을 주기적으로 재학습시킬 때 사용합니다.


## 1. 결정 - S3 (data lake)가 꼭 필요한 이유

현재 방식(`logs.json` 파일에 계속 텍스트를 이어 붙이는 방식)을 그대로 유지할 경우 다음과 같은 문제가 발생합니다.

- **서버 용량 부족:** `로그 파일`이 무한정 커지다가 결국 서버의 하드디스크 용량을 다 차지해 서버가 뻗어버립니다.
- **데이터 유실 위험:** 서버가 갑자기 꺼지거나 날아가면, 파일에 적혀있던 로그 데이터도 함께 날아갑니다.
- **확장성 문제:** 나중에 사용자가 많아져서 백엔드 서버를 2대, 3대로 늘리면, 각 서버마다 `logs.json` 파일이 따로 생겨서 하나로 합쳐서 보기 매우 힘들어집니다.

> 즉, "서버에 파일로 남긴다"는 현재 상태에 머무르면 안 되고, "이 파일을 주기적으로 안전한 곳(S3)으로 빼낸다"는 작업이 반드시 필요합니다.

#### Apache Kafka (실시간 대용량 스트리밍의 사실상 표준 / 엔터프라이즈급)

쿠팡, 넷플릭스, 토스 등 대용량 실시간 처리가 필요한 모든 IT 기업이 사용하는 끝판왕 데이터 파이프라인

- **동작 방식:** 백엔드 서버(Producer)가 이벤트 로그를 Kafka의 'Topic'이라는 공간에 쏘면, 이 데이터를 필요로 하는 **여러 시스템(Consumer - 추천 엔진, 통계 서버, S3 백업 워커 등)**이 각자의 처리 속도에 맞춰 독립적으로 실시간 데이터를 가져갑니다.
- **장점:**
    
    1. **압도적인 처리량:** 초당 수십만~수백만 건의 트래픽이 몰려도 서버가 뻗지 않고 든든하게 방어(Buffer)해 줍니다.
    
    2. **완벽한 확장성(Pub/Sub):** 한 번 수집한 데이터를 A팀, B팀, C팀이 동시에 가져가도 시스템에 무리가 없습니다.
    
    3. **데이터 복구(Replay):** Redis와 달리 데이터를 하드디스크에 안전하게 보관하므로, 추천 엔진 서버가 잠깐 죽었다가 살아나도 못 받은 데이터부터 다시 읽어올 수 있습니다.
    
- **단점:** 설정, 운영, 유지보수 난이도가 극악에 가깝습니다.
    - 제대로 띄우려면 서버가 최소 3대(클러스터) 이상 필요하고
    - 메모리도 많이 먹기 때문에,
    - 소규모 프로젝트나 1인 개발에서는 인프라 관리하다가 지칠 수 있습니다.
